# AIWiki 运营反馈闭环

AIWiki 当前采用低反馈阶段的运营方式。群里反馈少，不等于应该立刻做更多功能；更合理的处理是把试用路径、案例和提问方式做得更清楚。

这份文档把公开试用反馈、沉默信号、月度路线图复盘和开发队列连接起来。

## 信号分类

先分类，再决定是否进入开发队列。

| 分类 | 含义 | 常见处理 |
| --- | --- | --- |
| 安装 | Node.js、npm、全局命令、助手重启或路径设置失败。 | 优先改安装提示、FAQ 或 `doctor` 引导。 |
| 首次使用 | 用户安装后不知道下一步。 | 优先改 Quick Start、使用指南、案例或样例。 |
| 入库结果 | 第一份资料没有变成能理解的文件。 | 优先改 Agent handoff、案例、生成文件说明或诊断。 |
| 目录理解 | 用户找不到或不信任生成文件。 | 优先改文档、dashboard 示例、status/lint 输出或命名说明。 |
| 查询和复用 | `query` / `context` 找到内容，但用户不知道怎么用。 | 优先改复用说明、案例和助手指导。 |
| 功能请求 | 用户请求爬取、多知识库、向量检索、RBAC、RSS 等扩展。 | 先进入 keep / defer / Pro / reject 分类，不直接进入基础版队列。 |

## 进入开发队列的标准

群反馈不会直接变成功能。候选需求进入基础版 AIWiki 队列前，至少满足两项：

- 提升首次使用成功率；
- 降低用户理解成本；
- 提升本地知识复用；
- 让宿主 Agent 更稳定地调用 AIWiki 命令；
- 减少重复支持问题；
- 保持基础版产品边界。

候选项还必须写清：

- 用户问题；
- 证据来源；
- 影响的工作流步骤；
- 验收标准；
- 明确的范围外检查；
- 如果需要发布包，要写远端验证计划。

## 低反馈阶段怎么推进

当群反馈较少时：

- 自己跑一个小试用任务，并发布完整步骤；
- 发布一个 `examples/public-trial-scenarios/` 里的场景；
- 问一个窄问题，不问泛泛的“大家觉得怎么样”；
- 按试用步骤记录沉默，不把沉默解释成想象中的功能需求；
- 先改下一次试用提示，再考虑新增 CLI 能力；
- 没有证据时保留 `no-change` 判断。

不要把沉默理解为可以直接增加爬虫、向量检索、浏览器插件、多知识库、RBAC、RSS、定时采集或遥测。

## 遥测边界

这个运营闭环不向 AIWiki CLI 添加遥测。

如果未来需要遥测，必须作为单独 opt-in 任务设计，写清用户同意、采集字段、存储、保留周期、关闭方式和文档。在此之前，反馈只通过模板、issue、群讨论和人工试用笔记收集。
